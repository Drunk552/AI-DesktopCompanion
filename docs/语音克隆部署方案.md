# 语音克隆部署方案文档

## 一、概述

本文档描述 AI-DesktopCompanion 项目的语音克隆（Voice Cloning）部署方案。

目标：让 AI 宠物使用训练后的特定人声（如侯明昊）进行语音输出，实现"听到偶像声音和你对话"的体验。

### 核心原则

- 训练与推理分离：训练在高性能 PC 上完成，推理在云端/本地执行
- 多层降级策略：优先本地预录 → 云端 GPU 推理 → 云端 CPU 兜底 → 本地轻量 TTS
- 零成本优先：所有方案均可免费运行，适合学生群体

### 整体数据流

```
用户说话 → STT语音识别(树莓派本地) → 文本
                                         │
                                         ▼
                                  Gemini 3.0 API(AI推理)
                                         │
                                         ▼
                                    AI回复文本
                                         │
                              ┌──────────┴──────────┐
                              │                     │
                        匹配预录短句            非常规回复
                              │                     │
                        本地播放.wav         云端TTS API生成
                         (0延迟)           (1-15秒延迟)
                              │                     │
                              └──────────┬──────────┘
                                         │
                                         ▼
                                     扬声器播放
```

## 四、语音克隆TTS — 三层降级策略

### 第一层：本地预录短句（0延迟，¥0）

在 PC 上用 GPT-SoVITS 训练好侯明昊声音模型后，批量预生成常用短句音频。

#### 预录策略

```
4种情绪 × 5种好感度 = 20种状态组合

每种组合预录 20-25 条典型回复：
  平静+陌生: "嗯""哦""随便""都行""不知道"
  平静+亲密: "你在干嘛呀～""今天怎么样""想你了嘛"
  开心+熟悉: "哈哈还好啦""你也很开心嘛""嗯嗯好呀"
  难过+亲近: "怎么了""谁欺负你了""别难过了"
  生气+疏离: "哦""没事""不用管我"
  ...

20种组合 × 25条 = 500条预录音频
500条 × ~50KB/条 ≈ 25MB 存储空间
覆盖约 60-70% 的对话场景
```

#### 目录结构（规划）

```
ai_pet/assets/voices/
├── calm_stranger/        # 平静+陌生
│   ├── 001_嗯.wav
│   ├── 002_哦.wav
│   └── ...
├── calm_intimate/        # 平静+亲密
│   ├── 001_你在干嘛呀.wav
│   └── ...
├── happy_familiar/       # 开心+熟悉
├── sad_close/            # 难过+亲近
├── angry_distant/        # 生气+疏离
└── ...                   # 其余15种组合
```

#### 匹配逻辑

```
AI回复文本 → 模糊匹配预录库
  ├── 完全匹配 → 直接播放
  ├── 包含匹配 → 播放最接近的短句
  └── 无匹配 → 进入第二层（云端TTS）
```

**Cell 5：创建 cloudflared 隧道**

```python
!wget https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64.deb
!dpkg -i cloudflared-linux-amd64.deb

import subprocess, time, re

proc = subprocess.Popen(
    ['cloudflared', 'tunnel', '--url', 'http://localhost:5000'],
    stdout=subprocess.PIPE, stderr=subprocess.STDOUT
)

time.sleep(5)

for line in proc.stdout:
    line = line.decode('utf-8')
    if 'trycloudflare.com' in line:
        url = re.search(r'https://[a-z0-9\-]+\.trycloudflare\.com', line)
        if url:
            print(f"TTS API 地址: {url.group()}")
            break
```

#### 选择 cloudflared 而非 ngrok 的原因

| 对比项 | ngrok | cloudflared |
|--------|-------|-------------|
| 费用 | 免费版有限制 | 完全免费无限制 |
| 注册 | 需要注册+authtoken | 不需要注册 |
| URL稳定性 | 每次重启变化 | 每次重启变化 |
| 速度 | 快 | 快 |

#### URL 变化问题

cloudflared 每次重启 Colab 后 URL 会变化。解决方案：

**方案A：自动通知树莓派（推荐）**

```
Colab 启动 → 获取新 URL → POST 通知树莓派

树莓派端运行一个配置更新接口：
  POST /update_tts_url
  Body: { "url": "https://new-url.trycloudflare.com" }

Colab Cell 5 中加入：
  requests.post("http://树莓派IP:8080/update_tts_url",
                json={"url": new_url})
```

**方案B：手动更新配置文件**

```
Colab 启动后 → 控制台显示新 URL → 手动修改树莓派配置
适合每天只启动一次的场景
```

#### Colab 免费版限制

| 限制项 | 内容 | 应对策略 |

### 第三层：Hugging Face Spaces CPU 兜底（5-15秒延迟，¥0）— 24小时可用

当 Colab 不可用时（断连/未启动），使用 HF Spaces 作为兜底。

#### 资源规格

| 项目 | 说明 |
|------|------|
| CPU | 2 vCPU |
| RAM | 16 GB |
| 存储 | 50 GB |
| 费用 | ¥0（永久免费） |
| 运行时长 | 24小时（需保活） |

#### 纯 CPU 推理速度

| 文本长度 | 生成时间 |
|---------|---------|
| 短句(10字) | 5-10秒 |
| 中句(20字) | 10-20秒 |
| 长句(30字+) | 15-30秒 |

#### 部署步骤

1. 注册 huggingface.co（免费，无需实名）
2. 新建 Space → 选择 Docker SDK
3. 编写 Dockerfile 部署 GPT-SoVITS + FastAPI
4. 上传训练好的侯明昊模型权重
5. 树莓派通过 `https://xxx.hf.space/tts` 调用 API

#### 48小时休眠问题

HF Spaces 48小时无访问会自动休眠。使用 UptimeRobot 保活：

1. 注册 uptimerobot.com（免费版支持 50 个监控项）
2. 添加 HTTP 监控，每 30 分钟访问一次 Space 域名
3. Space 永远不会休眠

## 五、AI 推理方案：Gemini 3.0 API

### 替换现有 Ollama 调用

当前代码使用 Ollama 本地推理（gemma.h/cpp），计划替换为 Gemini 3.0 API。

| 对比项 | Ollama (当前) | Gemini 3.0 (计划) |
|--------|--------------|-------------------|
| 运行位置 | 树莓派本地 | Google 云端 |
| 内存占用 | 2-4 GB | 0（树莓派端） |
| 模型能力 | Gemma 4 (小模型) | Gemini 3.0 Pro (大模型) |
| 响应速度 | 较慢 | 较快 |
| 费用 | ¥0 | ¥0（已有 token） |
| 网络要求 | 不需要 | 需要访问 Google API |

### API 调用变更

```
当前：POST http://localhost:11434/api/chat
改后：POST https://generativelanguage.googleapis.com/v1beta/models/gemini-3.0-pro:generateContent?key=YOUR_TOKEN
```

### 树莓派访问 Google API

树莓派调用 Gemini API 需要能访问 Google 服务。可选方案：

- 树莓派配置代理（科学上网客户端）
- 通过 Cloudflare Worker 中转（无需树莓派科学上网）
- 自建反向代理

## 六、训练流程（PC 端一次性完成）

### 训练工具：GPT-SoVITS

| 项目 | 说明 |
|------|------|
| 训练数据 | 侯明昊音频素材（1分钟以上高质量录音） |
| 训练平台 | PC 端（需 NVIDIA GPU，6GB+ VRAM） |
| 训练时长 | 约 1-4 小时 |
| 输出 | GPT 权重 + SoVITS 权重（约 500MB-1GB） |

### 训练步骤

1. 收集侯明昊音频素材（访谈、节目片段等，尽量干净无背景音）
2. 使用 GPT-SoVITS WebUI 进行数据预处理（UVR5 人声分离 + ASR 标注）
3. 微调训练 SoVITS 模型
4. 微调训练 GPT 模型
5. 测试推理效果，调整参数
6. 导出模型权重文件
7. 将权重文件上传至 Google Drive 和 Hugging Face

### 预录制步骤

训练完成后，在 PC 上批量生成预录短句：

```
1. 加载训练好的侯明昊模型
2. 读取预录文本列表（500条短句）
3. 逐条生成语音并保存为 .wav
4. 按情绪×好感度分类存放
5. 拷贝到树莓派 assets/voices/ 目录
```

## 七、日常使用流程

### 场景一：完整体验（电脑旁）

```
1. 打开电脑浏览器 → 访问 Google Colab → 运行 Notebook
2. 等待 2-3 分钟（安装依赖 + 加载模型 + 启动隧道）
3. Colab 输出 TTS API URL → 树莓派自动/手动更新配置
4. 开始聊天，享受 GPU 级 TTS 速度（1-3秒响应）
5. 聊完关闭 Colab 页面
6. 树莓派自动降级到预录短句模式
```

### 场景二：不在电脑旁

```
1. HF Spaces 24小时运行（UptimeRobot 保活）
2. 树莓派检测到 Colab 不可用 → 自动切换到 HF Spaces
3. TTS 延迟增加（5-15秒），但功能正常
4. 常用短句仍由本地预录覆盖（0延迟）
```

### 场景三：全部云端不可用

```
1. 树莓派使用 Piper TTS 本地生成语音
2. 声音不是侯明昊的，但至少能说话
3. 或仅显示文字不播放语音
```
