# UI 像素宠物动画与聊天背景方案文档

## 一、概述

本文档描述 AI-DesktopCompanion 项目的两项 UI 增强方案：

1. **像素宠物动画** — 将当前静态文本占位 `(·ω·)` 替换为帧动画宠物形象
2. **聊天背景自定义** — 左侧宠物面板和右侧聊天面板支持更换背景图片

两项方案均针对树莓派 4B/5B 优化，在有限硬件资源下实现最佳视觉效果。

当前宠物形象方案已收敛为：**像素化线条小狗**。

- 默认角色：白色线条小狗
- 可选角色：黄色线条小狗
- 两者使用相同动作结构和人格逻辑，仅在配色和气质细节上区分

## 三、像素宠物动画方案

### 3.1 方案选型

### 3.1A 宠物角色定位

本项目中的宠物定义为：**陪伴型像素化“线条小狗”**。

选择它的原因不是单纯因为形象流行，而是它天然适合这个项目的 UI 气质：轮廓简洁、识别度高、可爱但不幼稚、容易承载温柔陪伴型人格。

它不是单纯的状态图标，也不是高频抢眼的游戏角色，而是一个长期驻留在 UI 左侧、能够通过表情和小动作体现“陪伴感”和“关系反馈”的拟人化伙伴。

角色关键词：

- 线条简洁
- 温柔体贴
- 理解人情绪
- 安静陪伴
- 拟人但不过度戏剧化

人格参考方向：

- 可参考 Gemini 3 对话风格中的温和、耐心、理解人性的陪伴感
- 说话和反馈气质应偏温柔、稳定、不过分夸张
- 面对用户情绪时优先表现理解、接纳、安抚，而不是强烈评判或过度卖萌

实现策略：

- 前期先写死一套固定人格，作为默认宠物性格
- 该人格以“温柔体贴、能共情、能安抚、轻度拟人化依恋”为核心
- 后续若需要扩展为“可切换角色人格”，再单独设计角色配置体系

参考图结论：

- 线条小狗的核心优势不是细节丰富，而是“极简轮廓 + 极少五官 + 很准的情绪姿态”
- 角色识别主要依赖头部轮廓、耳朵形状、点状眼睛、鼻嘴连线和身体整体姿势
- 情绪传达主要依赖嘴型、手势、耳朵朝向、身体倾斜和少量辅助符号
- 因此本项目不应追求复杂动画，而应追求“少帧数、高识别、神态稳定”的桌宠表达

皮肤策略：

- 默认加载白色线条小狗 `white`
- 预留黄色线条小狗 `yellow` 作为用户可选皮肤
- 两套皮肤共用同一情绪状态机和动画资源结构
- UI 层只需要切换当前角色资源组，不需要改动情绪判定逻辑

拟人化目标：

- 宠物的情绪变化不只来自系统事件，也来自“主人当前表情”和“对话态度”
- 它的反应逻辑参考亲密关系中的情绪反馈，而不是简单的功能提示
- 当主人表现出温和、开心、安抚、夸奖等态度时，宠物会更放松、更亲近、更开心
- 当主人持续表现出烦躁、愤怒、攻击性表达时，宠物会表现出委屈、低落、不安或轻微闹别扭

设计边界：

- 情绪表达要可爱、克制，避免过强负反馈造成用户反感
- “生气”更接近闹别扭或受伤，不建议做成攻击性很强的表现
- 宠物始终应保持可被安抚、可恢复亲近的状态，符合长期陪伴产品定位

| 方案 | 实现方式 | 内存占用 | CPU占用 | 视觉效果 | 推荐度 |
|------|---------|---------|---------|---------|-------|
| **LVGL AnimImage** | 帧序列动画 | 低 | 极低 | ★★★★ | ⭐⭐⭐⭐⭐ |
| LVGL GIF | 解码 GIF 文件 | 中 | 中 | ★★★ | ⭐⭐⭐ |
| Canvas 逐帧绘制 | 手动绘制像素 | 低 | 高 | ★★★★★ | ⭐⭐ |

**推荐方案：LVGL AnimImage（帧序列动画）**

原因：
- `lv_conf.h` 已启用 `LV_USE_ANIMIMG 1`，无需修改配置
- 帧数据为预渲染的 C 数组，零解码开销
- CPU 占用极低，适合树莓派
- 可精确控制帧率、循环、切换

### 3.2 帧动画规格设计

#### 基本参数

```
单帧尺寸: 112 × 112 像素
色深: ARGB8888（32位，与 LV_COLOR_DEPTH 32 一致）
帧数: 每组动画 4-8 帧
帧率: 8-12 FPS（像素画不需要高帧率）
```

制作建议：

- 当前确认规格为 `112×112`
- 该尺寸更适合当前 `192px` 左侧面板宽度，能让宠物区域更饱满
- 后续新增状态动画和皮肤时，统一按 `112×112` 制作，避免混用尺寸

#### 动画状态组（对应情绪策略矩阵）

根据 `strategy.h` 中的 4 种情绪设计 4 组动画：

| 动画组 | 情绪 | 帧数 | 描述 |
|--------|------|------|------|
| idle | 平静 | 4帧 | 微微呼吸/眨眼 |
| happy | 开心 | 6帧 | 蹦跳/摇尾巴/星星眼 |
| sad | 难过 | 4帧 | 低头/流泪/耷拉 |
| angry | 生气 | 6帧 | 跺脚/冒烟/红脸 |

角色皮肤维度建议：

| 皮肤 | 说明 | 默认状态 | 用户可选 |
|------|------|---------|---------|
| white | 白色线条小狗 | 是 | 是 |
| yellow | 黄色线条小狗 | 否 | 是 |

每套皮肤都需要完整提供 `idle/happy/sad/angry` 四组动画，保证用户切换角色皮肤后不会出现缺失状态。

建议进一步细化为“基础情绪 + 关系反馈”表现：

- `idle`：默认陪伴态，轻微呼吸、眨眼、耳朵轻动、身体小幅晃动
- `happy`：被夸奖、被安抚、被喜欢时的积极反馈，表现为靠近感、摇尾巴、眼神发亮、短暂蹦跳
- `sad`：主人语气差、忽视、连续负面互动后的低落反馈，表现为低头、耳朵下垂、身体缩起、尾巴下落
- `angry`：主人明显烦躁或攻击性表达时的闹别扭反馈，表现为撇头、尾巴甩动、轻微鼓气或短暂别过脸

不建议把 `angry` 做成真正敌对状态，更适合表达为“我有点难过 / 我不开心了”，这样更符合温柔陪伴型产品气质。

参考图对应建议：

- 抱爱心、贴近、挥手、轻跳，这类姿态非常适合用于 `happy`
- 站立、轻微晃动、抬手打招呼，这类姿态适合用于 `idle`
- 低头、缩起、别过脸、耳朵下压，更适合用于 `sad` 或 `angry`
- 表情包中出现的小爱心、音符等可以作为辅助元素，但只建议在触发动画中短暂出现

#### 情绪触发原则

宠物状态建议同时参考两类输入：

- 主人表情识别结果：开心、平静、疲惫、难过、愤怒等
- 对话态度判断：夸奖、安抚、亲近、冷淡、抱怨、攻击性表达等

推荐逻辑：

- 主人表情决定宠物的基础共情方向
- 对话态度决定宠物的即时反馈强度
- 当两者一致时，宠物情绪变化更明显
- 当两者冲突时，以“对话态度”优先，因为这是对宠物的直接互动信号
- 默认人格会影响反馈方式，即同样是负面输入，也优先表现为委屈、失落、安静疏离，而不是激烈反击

示例：

- 主人开心且对话温和：宠物进入 `happy`
- 主人平静但语气温柔：宠物维持 `idle` 并短暂播放 `happy` 反馈
- 主人生气且言语激烈：宠物进入 `sad` 或 `angry`
- 主人先前态度较差，随后安抚或哄宠物：先短暂 `sad`，再缓慢恢复到 `happy`
- 主人情绪低落但表达平和：宠物不必同步低落到底，可表现为安静陪伴式 `idle` 或带安抚感的轻度 `happy`

#### 动画切换原则

- `idle` 必须长期可循环，且不影响聊天阅读
- `happy/sad/angry` 建议播放 1-2 轮后自动回到 `idle`
- 短时间内重复识别到相同情绪时，不要重复强触发，避免界面过于躁动
- 情绪恢复应有缓冲，不建议每次识别变化都瞬间切换
- 主体锚点保持一致，避免切换动画时出现位置抖动

#### 单帧内存计算

```
单帧大小 = 112 × 112 × 4 bytes (ARGB8888)
         = 50,176 bytes
         ≈ 49 KB

一组动画(8帧) ≈ 49 KB × 8 ≈ 392 KB
四组动画总计   ≈ 392 KB × 4 ≈ 1568 KB ≈ 1.53 MB

两套皮肤总计   ≈ 1.53 MB × 2 ≈ 3.06 MB
```

LVGL AnimImage 使用 `lv_img_dsc_t` 数组描述帧序列：

static const uint8_t pet_idle_0_map[] = {

    // 由像素画工具导出
    ...

static const lv_img_dsc_t pet_idle_0 = {
    .header = {

        .w = 112,
        .h = 112,

### 3.5 情绪切换动画

当情绪更新时，切换动画组：

```cpp
void UIManager::switchPetAnimation(Emotion emotion) {
    switch (emotion) {
        case Emotion::HAPPY:
            lv_animimg_set_src(petAnimImg_, (const lv_img_dsc_t**)pet_happy_frames, 6);
            lv_animimg_set_duration(petAnimImg_, 600);
            break;
        case Emotion::SAD:
            lv_animimg_set_src(petAnimImg_, (const lv_img_dsc_t**)pet_sad_frames, 4);
            lv_animimg_set_duration(petAnimImg_, 800);
            break;
        case Emotion::ANGRY:
            lv_animimg_set_src(petAnimImg_, (const lv_img_dsc_t**)pet_angry_frames, 6);
            lv_animimg_set_duration(petAnimImg_, 500);
            break;
        default: // CALM
            lv_animimg_set_src(petAnimImg_, (const lv_img_dsc_t**)pet_idle_frames, 4);
            lv_animimg_set_duration(petAnimImg_, 500);
            break;
    }
    lv_animimg_start(petAnimImg_);
}
```

若后续增加角色皮肤选择，建议在情绪切换之前先确定当前皮肤资源组，例如：

- `white + happy -> dog_white_happy_frames`
- `yellow + happy -> dog_yellow_happy_frames`

这样可以保证“皮肤选择”和“情绪切换”两套逻辑互相独立。

### 3.6 像素画素材制作

推荐工具：

| 工具 | 平台 | 特点 |
|------|------|------|
| Aseprite | Win/Mac/Linux | 专业像素画编辑器，支持动画帧编辑和导出 |
| Piskel | Web(免费) | 在线像素画工具，支持动画预览 |
| LibreSprite | Win/Mac/Linux(免费) | Aseprite 开源分支 |

制作流程：

```
1. 在像素画工具中创建 112×112 画布
2. 逐帧绘制宠物动画（4-8帧/组）
3. 导出每帧为 PNG 文件
4. 使用 Python 脚本将 PNG 转换为 C 数组（lv_img_dsc_t 格式）
5. 将生成的 .c/.h 文件加入项目编译
```

建议的资源制作顺序：

1. 先完成白色线条小狗 `idle`
2. 再完成白色 `happy/sad/angry`
3. 白色版本验证通过后，再产出黄色版本
4. 黄色版本尽量复用相同动作模板，只调整配色和局部气质细节

配套美术资源制作规范可参考：`docs/线条小狗美术资源制作规范.md`

对于不会系统绘画的开发者，建议采用以下低门槛路线：

1. 先选定 1 张白色线条小狗站姿参考图，做默认基准形象
2. 在像素工具中临摹出简化版轮廓，而不是从零原创设计
3. 后续动画帧只微调嘴巴、眼睛、耳朵、手势和上下位移，不重画整只狗
4. 黄色版本在白色版本基础上换色和微调，不单独重做整套动作

这样可以把“会不会画画”的门槛降到最低，更适合当前项目推进。

PNG 转 C 数组的脚本思路：

```python
from PIL import Image
import sys

def png_to_lvgl_c(png_path, var_name, output_path):
    img = Image.open(png_path).convert('RGBA')
    w, h = img.size
    pixels = img.tobytes()  # RGBA 格式

    # LVGL ARGB8888 需要 A,R,G,B 顺序（不是 RGBA）
    argb_data = bytearray()
    for i in range(0, len(pixels), 4):
        r, g, b, a = pixels[i], pixels[i+1], pixels[i+2], pixels[i+3]
        argb_data.extend([a, r, g, b])

    with open(output_path, 'w') as f:
        f.write(f'static const uint8_t {var_name}_map[] = {{\n')
        for i in range(0, len(argb_data), 16):
            line = ', '.join(f'0x{b:02X}' for b in argb_data[i:i+16])
            f.write(f'    {line},\n')
        f.write('};\n\n')
        f.write(f'static const lv_img_dsc_t {var_name} = {{\n')
        f.write(f'    .header.magic = LV_IMAGE_HEADER_MAGIC,\n')
        f.write(f'    .header.cf = LV_COLOR_FORMAT_ARGB8888,\n')
        f.write(f'    .header.w = {w},\n')
        f.write(f'    .header.h = {h},\n')
        f.write(f'    .data_size = {len(argb_data)},\n')
        f.write(f'    .data = {var_name}_map,\n')
        f.write('};\n')
```

### 3.7 美术规范

为保证像素化线条小狗在多组情绪动画中保持统一观感，建议增加以下资源规范：

#### 角色外形规范

- 角色主体为圆润轮廓的线条小狗，避免尖锐、攻击性过强的造型
- 头身比例建议偏可爱方向，可采用 1.2:1 到 1.5:1 的大头短身比例
- 轮廓识别优先级高于内部细节，缩小后也应一眼看出是“线条小狗”
- 耳朵、尾巴、脸部五官是主要识别点，动作变化应优先体现在这些部位

#### 像素规格规范

- 单帧画布统一为 `112×112`
- 角色主体建议控制在 `48×48` 到 `56×56` 可视范围内
- 四周预留 `4-8px` 安全边距，避免动作贴边
- 所有动画帧使用同一锚点，建议以脚底中心或身体中心作为对齐基准

#### 调色规范

- 每只宠物主色建议控制在 `4-6` 个核心颜色内，不含透明色
- 明暗层级建议控制在 `3` 层以内，避免像素画过脏
- 高光颜色只用于眼睛、鼻尖、铃铛、饰品等小面积点缀
- `happy` 可增加轻微高亮；`sad/angry` 主要通过姿态和五官变化表达，不建议整只狗大幅变色

#### 线条与细节规范

- 描边粗细保持统一，避免某些帧轮廓粗细变化造成闪烁感
- 面部表情优先用眼睛、耳朵、嘴型变化表达，不依赖复杂特效
- 除非明确需要，不建议大量使用星星、火焰、泪滴等外挂特效
- 特效若存在，面积不应超过角色主体的 `20%`，避免喧宾夺主

#### 动作规范

- `idle` 以呼吸、眨眼、耳朵轻抖、尾巴微摆为主
- `happy` 以轻跳、前倾、尾巴摇动、脸部发亮感为主
- `sad` 以低头、缩肩、耳朵下垂、尾巴下落为主
- `angry` 以撇头、炸毛一点点、尾巴甩动为主，不做攻击动作
- 动作幅度控制在小范围内，避免像游戏角色一样频繁大位移

#### 资源命名规范

- 动画帧命名统一为：`pet_<state>_<index>.png`
- 例如：`pet_idle_0.png`、`pet_happy_3.png`
- 导出的 C 数组和描述符命名与源文件保持一致，降低接入成本
- 若后续支持多皮肤，建议扩展为：`cat01_idle_0.png`

#### 背景搭配规范

- 宠物背景饱和度应低于宠物主体，避免主次不分
- 聊天背景应优先低对比、低纹理，不干扰文字阅读
- 若线条小狗为浅色系，背景建议使用中深色；若角色轮廓较深，背景建议使用浅灰或低饱和暖色
- 背景和宠物不应共享过于接近的主色，否则边界会融在一起

## 四、聊天背景自定义方案

### 4.1 功能描述

- 右侧聊天面板：支持更换背景图片（类似微信聊天背景）
- 左侧宠物面板：支持更换背景图片
- 提供内置背景图库供选择
- 背景图存储在 SD 卡指定目录

### 4.2 背景图规格

#### 背景铺满策略

- 左侧背景建议铺满整个左侧面板区域
- 右侧背景建议铺满整个右侧面板区域
- 但左右两侧的背景设计策略不能完全相同

左侧背景设计要求：

- 允许有一定场景感和陪伴氛围
- 可以使用房间、夜灯、窗边、草地、像素小屋等轻场景主题
- 中间区域应尽量留出宠物主体展示空间
- 右上角应避免复杂高对比元素，给摄像头小窗留视觉缓冲区
- 底部中间应避免文字密集或高亮元素，给宠物名字标签留空间

右侧背景设计要求：

- 必须以不影响聊天阅读为第一优先级
- 优先使用低对比、低纹理、低饱和背景
- 可以有轻渐变、轻纹理、轻噪点，但不能有复杂主体图案
- 不建议使用强像素花纹、大面积插画、强对比装饰
- 顶部状态栏和底部输入区所在区域应尽量保持干净

#### 右侧聊天面板

```
尺寸: 288 × 320 像素
色深: ARGB8888 (32位)
单张大小: 288 × 320 × 4 = 368,640 bytes ≈ 360 KB
```

#### 左侧宠物面板

```
尺寸: 192 × 320 像素
色深: ARGB8888 (32位)
单张大小: 192 × 320 × 4 = 245,760 bytes ≈ 240 KB
```

推荐实际出图尺寸：

- 左侧默认背景：`192 × 320`
- 右侧默认背景：`288 × 320`

说明：

- 当前项目界面尺寸固定为 `480 × 320`
- 左右背景直接按面板实际尺寸出图，接入最简单，显示结果也最稳定
- 如后续需要做缩放动画或多端适配，再考虑更高分辨率版本

#### 两张背景合计

```
360 KB + 240 KB = 600 KB ≈ 0.6 MB（运行时内存）
```

### 4.3 背景图优化策略

为降低内存占用，可采用以下优化：

#### 策略一：使用 RGB565 格式（节省50%内存）

```
ARGB8888: 288 × 320 × 4 = 360 KB
RGB565:   288 × 320 × 2 = 180 KB

两张合计: 180 KB + 120 KB = 300 KB

注意: RGB565 不支持透明度，背景图通常不需要透明度
```

#### 策略二：降低分辨率 + 拉伸

```
聊天背景实际尺寸 288×320，但可以存储 144×160 的半分辨率图
显示时由 LVGL 自动拉伸

144 × 160 × 2 (RGB565) = 46 KB / 张
两张合计: ~92 KB

代价: 细节模糊，但聊天背景通常是纹理/渐变，影响不大
```

#### 策略三：纯色 + 渐变（零图片内存）

```
使用 LVGL 内置渐变样式：
lv_obj_set_style_bg_grad(dir, stop1, stop2)

内存: 0
效果: 简洁但缺乏个性
```

### 4.4 推荐方案：RGB565 全分辨率

在树莓派 4B 上，600 KB 的背景图内存完全可接受：

```
树莓派 4B 2GB 总内存
- 系统占用:        ~400 MB
- UI + 帧缓冲:     ~0.6 MB
- OpenCV:          ~100 MB
- STT:             ~180 MB
- 背景图:          ~0.6 MB
- 预录音频:        ~25 MB
- 剩余:            ~1.3 GB

0.6 MB 背景图占总内存 0.03%，完全可忽略
```

### 4.5 实现代码思路

#### 背景图加载

```cpp
// ui.h 新增成员
lv_obj_t* leftBgImage_ = nullptr;
lv_obj_t* rightBgImage_ = nullptr;
std::string currentLeftBg_;
std::string currentRightBg_;
```

#### 创建背景图层

在 `createLeftPanel()` 和 `createRightPanel()` 中，将背景图作为第一个子对象（最底层）：

```cpp
void UIManager::createLeftPanel(lv_obj_t* parent) {
    // ... 创建 leftPanel_ ...

    // 背景图（最底层）
    leftBgImage_ = lv_image_create(leftPanel_);
    lv_image_set_src(leftBgImage_, "/path/to/default_left_bg.bin");
    lv_obj_set_size(leftBgImage_, 192, 320);
    lv_obj_set_pos(leftBgImage_, 0, 0);
    lv_obj_add_flag(leftBgImage_, LV_OBJ_FLAG_HIDDEN); // 默认隐藏，用纯色

    // ... 其余子对象（宠物动画、摄像头等）...
}
```

#### 切换背景

```cpp
void UIManager::setChatBackground(const std::string& bgName, bool isRight) {
    std::string path = "/opt/ai_pet/assets/backgrounds/" + bgName;

    if (isRight) {
        lv_image_set_src(rightBgImage_, path.c_str());
        lv_obj_clear_flag(rightBgImage_, LV_OBJ_FLAG_HIDDEN);
        currentRightBg_ = bgName;
    } else {
        lv_image_set_src(leftBgImage_, path.c_str());
        lv_obj_clear_flag(leftBgImage_, LV_OBJ_FLAG_HIDDEN);
        currentLeftBg_ = bgName;
    }
}

void UIManager::resetBackground(bool isRight) {
    if (isRight) {
        lv_obj_add_flag(rightBgImage_, LV_OBJ_FLAG_HIDDEN);
        currentRightBg_.clear();
    } else {
        lv_obj_add_flag(leftBgImage_, LV_OBJ_FLAG_HIDDEN);
        currentLeftBg_.clear();
    }
}
```

### 4.6 背景图文件格式

LVGL 支持通过文件系统加载图片（`LV_USE_FS_POSIX 1` 已启用）。

#### 方式一：LVGL 二进制格式（推荐）

使用 LVGL 官方工具将 PNG 转换为 `.bin` 格式：

```
转换工具: LVGL Image Converter (在线)
地址: https://lvgl.io/tools/imageconverter

设置:
  Color format: RGB565
  Output format: Binary (RGB565 Swap)
  选中 "Include header"

输出: .bin 文件，可直接被 lv_image_set_src() 加载
```

推荐理由：

- 更适合运行时加载背景图
- 比将背景编译进 `.c` 更方便后续替换素材
- 适合左侧和右侧各自维护多张候选背景

#### 方式二：C 数组编译嵌入

适合内置默认背景，与像素宠物帧数据一样编译进二进制：

```
优点: 不需要文件系统读取
缺点: 增加编译产物大小
适合: 1-2 张默认背景
```

#### 当前项目格式建议

- 左侧背景：优先使用 `PNG` 作为设计源文件，接入时转换为 LVGL `.bin`
- 右侧背景：优先使用 `PNG` 作为设计源文件，接入时转换为 LVGL `.bin`
- 不建议直接用 `JPG` 作为最终接入资源，尤其不适合边缘清晰的像素风素材
- 若只是前期快速预览，也可以暂时使用 `.c` 编译嵌入，但不建议长期这么维护背景图库

## 七、验收标准

### 7.1 视觉验收

- 在 `480×320` 屏幕下，用户能一眼识别左侧角色为“像素化线条小狗”
- `idle/happy/sad/angry` 四组动画在不看文字说明时也能大致分辨情绪差异
- 多帧播放过程中，角色主体位置稳定，无明显抖动、跳帧、错位
- 圆润、治愈、陪伴感的角色气质在四种状态下保持一致，不因情绪切换而风格断裂
- 聊天背景不会压过宠物主体和消息文本，不造成阅读负担

### 7.2 交互验收

- 宠物能根据主人的表情和对话态度做出符合预期的情绪反馈
- 当主人表现出温和、安抚、喜欢等态度时，宠物能进入更亲近、更开心的反馈状态
- 当主人表现出烦躁、责备、攻击性表达时，宠物会表现出不开心、委屈或闹别扭，而不是无反应
- 情绪切换不过于频繁，同类情绪不会在短时间内重复疯狂触发
- 强情绪结束后，宠物能在合理时间内恢复到 `idle`

### 7.3 性能验收

- 宠物动画连续播放 `10` 分钟以上无明显卡顿、闪烁或异常停播
- 背景图加载和切换后，聊天列表滚动、消息刷新不出现肉眼可见掉帧
- 在目标树莓派设备上，新增动画和背景方案后内存占用保持在预估范围内
- 动画方案不引入额外高频解码开销，CPU 占用无异常升高

### 7.4 资源验收

- 所有动画帧尺寸、命名、锚点、色彩规范一致
- 导出的资源文件能稳定接入 LVGL，不出现格式不兼容问题
- 默认背景和默认宠物动画能随程序正常发布，不依赖临时手工操作
- 至少准备一套完整可用的默认白色线条小狗主题资源，保证首次启动时就有统一视觉效果

### 7.5 主观体验验收

- 用户在静态待机时不会觉得宠物“太吵”或“太抢戏”
- 用户在发生情绪互动时，能感受到宠物像在“回应自己”，而不是机械切换图标
- 宠物整体观感更接近“陪伴型电子女友/伙伴”而不是工具型状态指示器
